## §1. Продукт

### 1.1 Для кого и зачем

Основной пользователь: русскоязычный, уровень A2–B1 в чтении, заметно слабее в активном построении фраз и говорении. Цель — **автоматизм**: увидел мысль по-русски → мгновенно собрал корректную английскую фразу, письменно и устно.

Приложение — публичное, бесплатное, без регистрации: открыл сайт (PWA), всё хранится локально в браузере, свой API-ключ подключается по желанию и даёт «репетитора».

**Реалистичный портрет аудитории (2026-07-08):** BYOK — фильтр; с ключом останутся в основном технически грамотные мотивированные самоучки (люди типа автора). Это осознанно: некоммерческий проект не обязан покрывать всех. Без ключа продукт — самоучитель со сверкой по эталонам (§1.4.3): достаточно для A1–A2, слабее к B2. Двуязычный UI и полировка онбординга не отменяются, но приоритезируются с оглядкой на этот портрет.

**Позиционирование одним абзацем («чем лучше English Dog», для README §18):** то же ядро — перевод по паттернам с ИИ-разбором, — но бесплатно, без подписки и сервера, с открытым кодом и данными, и с настоящим закреплением: после разбора ошибки юзер обязан воспроизвести фразу сам (REWRITE §6.1), а FSRS возвращает паттерн ровно тогда, когда тот начинает забываться (§10). *(Формулировка — черновик агента, юзер вправе переписать; но одна фраза позиционирования обязана существовать.)*

**Критерии успеха проекта** (личные, не телеметрия; черновик — уточнить юзеру): (а) автор сам занимается ≥4 недель подряд после Ф8; (б) ≥5 внешних пользователей дают фидбек через Issues; (в) A2→B1 у автора закрывается за прогнозируемый срок. По ним решается «продолжать ли Ф9+». Вопрос дистрибуции (как о проекте узнают) — решение юзера до Ф8; минимальный вариант: пост на Habr + r/languagelearning + тематические чаты.

**Честная граница возможностей (важно закрепить с самого начала, чтобы не переобещать):** тренажёр даёт автоматизм грамматики и лексики через перевод заданных предложений — это необходимое, но НЕ достаточное условие разговорной беглости. Он специально не тренирует: (а) порождение собственной мысли на английском без русской подсказки-стимула, (б) восприятие непредсказуемой естественной речи (акценты, редукции, темп), (в) диалог/переговоры смысла с живым собеседником. Это осознанный компромисс ради простоты и бесплатности, а не недосмотр. На онбординге и в README — честная фраза в духе «это тренажёр автоматизма, а не замена живому общению»; режимы «Про себя» (§8.4) и Speaking (§9.4) — мостики к свободной речи, но не полноценная замена разговорной практики.

### 1.2 Что приложение ДЕЛАЕТ

- Дриллы перевода RU→EN по паттернам (ядро продукта): 100+ предложений на конструкцию, подсказки, «сдаться», озвучка, мгновенная проверка.
- ИИ-репетитор: разбор ошибки в чате, наводящие вопросы, персональные добивающие примеры, генерация урока по запросу.
- **Свободный диалог** (§8.9): живой обмен репликами на английском на выбранную тему, без заданного русского предложения на перевод — тренирует порождение своей мысли и ведение разговора, а не только перевод чужой.
- Проверка «правильный, но другой перевод» — трёхуровневый каскад (локально → кэш → LLM-судья), см. §7.
- Интервальные повторения (FSRS) на уровне навыков, а не заучивания конкретных предложений (§10).
- Listening (диктант с эталонным TTS: американский/британский, мужской/женский голос) и Speaking (голосовой ввод — опционально, юзер выбирает голос или текст) (§9).
- Уровни A1–C1, placement-тест, быстрый скип модулей/уровней, экзамены модулей и уровней (§11).
- Работает офлайн после первой загрузки (кроме LLM-фич).

### 1.3 Что приложение НЕ делает (non-goals)

- Не учит отдельным словам в отрыве от предложений — никаких словарных карточек «word → перевод». Частотные списки используются только внутри пайплайна как фильтр сложности.
- Не пытается покрыть «весь английский» — только ~76 самых переносимых конструкций ядра A1–B2 (§3), плюс отдельный бэклог C1.
- Нет сервера, аккаунтов, подписок, платежей, сбора данных.
- Не генерирует основной контент LLM'ом в рантайме — базовые уроки собраны заранее из корпуса; LLM в рантайме только проверяет, объясняет и генерирует опциональные доп. уроки.
- Не готовит к конкретным экзаменам (IELTS/TOEFL).

### 1.4 Ключевые принципы

1. **Local-first**: весь прогресс в IndexedDB, экспорт/импорт бэкапа в JSON-файл.
2. **Корпус > генерация**: предложения из Tatoeba (проверенные людьми), LLM — добор дефицита офлайн-пайплайном и опциональные уроки по запросу.
3. **BYOK (bring your own key)**: Gemini (дефолт), OpenRouter, Groq, GigaChat, Yandex AI Studio, Ollama (локальный) — у провайдеров разная региональная доступность, поэтому адаптеров должно быть много: цепочка роутинга переживает недоступность любого из них (§8.2). Без ключа приложение работает как самоучитель со сверкой по эталонам: tier 1–2 + пословный диф + самооценка (§7). На A1–A2 диф решает почти всё, к B2 — заметно слабее судьи. В UI/README не называть это «полноценным» — честно: «ИИ добавляет оценку вариативных ответов и разбор».
4. **Бюджетная дисциплина LLM**: дневные счётчики, кэш принятых ответов, дешёвая модель для проверки, graceful degradation.
5. **Всё бесплатно**: статический хостинг (GitHub/Cloudflare Pages), free-tier LLM, открытые данные (CC-BY с атрибуцией).

## §2. Методологический фундамент

Каждая механика приложения привязана к подтверждённому принципу обучения — ничего «на рандом».

| Принцип (источник) | Механика в приложении |
|---|---|
| **Pattern practice / output-first**: активное построение фраз эффективнее пассивного узнавания (output hypothesis, Swain; практика English Dog/DadEnglish) | Ядро — письменный/устный перевод RU→EN, а не multiple choice. Ученик всегда строит предложение сам. |
| **Retrieval practice (testing effect)**: вспоминание укрепляет память сильнее перечитывания | Теория — 3–6 предложений, свёрнута по умолчанию. 95% времени — попытки вспомнить и построить. |
| **Spaced repetition**: интервальные повторения; FSRS — современный алгоритм, дефолт в Anki с 2023 | FSRS-карточки на уровне навыков + «пиявки» (§10). |
| **Interleaving**: перемешивание тем эффективнее блочной зубрёжки | В каждой сессии ~30% предложений из ранее пройденных навыков (§6.2). |
| **Comprehensible input, i+1** (Krashen; рекомендации ER Foundation: лёгкий приятный материал) | Лексика предложения не выше CEFR-уровня навыка (правило 95% покрытия, §4.5). Listening на своём уровне, со скоростью 0.8×/1×. |
| **Immediate corrective feedback**: мгновенная, короткая, по одной ошибке | Проверка сразу; объяснение ≤2 предложений; разбор одной ошибки за раз (§7, §8). |
| **Shadowing + ASR** для произношения (исследования 2025: связка shadowing и распознавания речи улучшает fluency/точность) | Speaking-режим: услышал → повторил → STT-транскрипт → диф с эталоном; авто-озвучка после правильного ответа. |
| **Nation, Four Strands**: баланс input / output / language focus / fluency | Дрилл (output+form) + listening (input) + review на скорость (fluency) + микротеория (form). |
| **Снижение тревожности через AI-собеседника** (мета-анализы 2025–2026: чат-боты дают наибольший прирост в speaking/writing) | Ошибки не наказываются: подсказки без штрафа, «сдаться» → перепечатать правильный ответ; тон репетитора тёплый, сократический (LearnLM/Study Mode-подход). |
| **Production effect** (сильные доказательства): произнесённое вслух запоминается на 10–20% лучше прочитанного про себя | «Вслух после верного» (§9.4), голосовые раунды, speaking-режимы. |
| **Self-reference effect** (сильные, d≈0.5–0.7): материал о себе запоминается лучше абстрактного | Режим «Про себя» (§8.4): свои предложения с паттерном идут в личные повторения. |
| **4/3/2 fluency** (Nation; сильные): повтор одного материала со сжимающимся временем строит беглость | Fluency-спринты на освоенном материале с уменьшающимся таймером (§6.3). |
| **Noticing** (сильные): явное сравнение своего вывода с образцом двигает усвоение | Пословный диф «мой ответ ↔ эталон» в каждом фидбеке + REWRITE (§6.1). |
| **Pretesting / generation effect** (умеренные): попытка до объяснения усиливает последующее усвоение | «Проба»: 2 предложения до теории при первом входе в навык (§3.3). |
| **Дозирование**: короткие ежедневные сессии эффективнее редких длинных | Сессия 15–25 минут (пресеты 5/15/25), дневная цель, streak с заморозкой (§12). |

Первоисточники для Codex (читать не обязательно, ссылки для README): englishprofile.org (EVP/EGP), learnenglish.britishcouncil.org, test-english.com, oxfordlearnersdictionaries.com (Oxford 3000/5000), erfoundation.org (extensive reading), github.com/open-spaced-repetition (FSRS), ai.google.dev (LearnLM/Gemini).

## §3. Учебная программа

### 3.1 Иерархия

```
Уровень (A1..C1)
└── Модуль (3–6 навыков + экзамен модуля)
    └── Навык / skill (= семейство паттернов, единица обучения и SRS)
        └── Предложения (60–150 шт., распределены по саб-дриллам)
```

Прогрессия: навыки модуля → экзамен модуля (≥80%) → следующий модуль. Уровень закрывается экзаменом уровня (§11). В настройках есть тумблер «свободный режим» — снять блокировки и ходить по карте свободно (автономия важнее жёсткости).

### 3.2 Анатомия навыка

Каждый навык содержит (формат пака — §4.8):

- `pattern` — формула («Do + you/we/they + V1 …?»);
- `theory_ru` — микротеория, 3–6 предложений простым языком, одна формула, два примера; целевая форма в примерах выделяется жирным (input enhancement — эффект слабый, но бесплатный). Генерируется пайплайном (§8.6), правится вручную при вычитке;
- `common_errors` — 3–5 типичных ошибок русскоязычных (для подсказок и репетитора);
- предложения с саб-типами: `affirm` (утверждение), `neg` (отрицание), `question` (общий вопрос), `wh` (спец. вопрос), `short_answer`, `mixed` (контраст с соседним паттерном);
- `probe_item_ids` — 2–3 репрезентативных предложения для placement/skip-тестов;
- `youglish_query` — фраза для виджета YouGlish.

Квоты предложений на навык: A1 — 100–150, A2 — 100–140, B1 — 80–120, B2 — 60–100, C1 — 40–80. Распределение саб-типов по умолчанию: affirm 40%, question 25%, neg 15%, wh 10%, mixed 10% (для вопросительных навыков — зеркально).

### 3.3 Структура дрилл-сессии (15–25 мин)

1. **Разминка**: 5–8 due-предложений из SRS-очереди (если есть).
2. **Проба** (pretesting, только при первом входе в навык): 2 предложения нового паттерна ДО показа теории — попытка «вслепую», без штрафа и без записи в статистику. Ошибка до объяснения усиливает последующее усвоение (generation/pretesting effect). UI обязан сказать одной строкой: «Попробуй наугад — не считается и помогает мозгу лучше усвоить правило» — иначе проба выглядит как «задача, которой не учили».
3. **Теория** (только при первом входе в навык, свёрнута повторно): формула + 2 примера.
4. **Основной дрилл**: 12–20 новых предложений навыка, сложность по нарастающей (`difficulty` 1→5).
5. **Interleave**: каждые ~3 предложения вставляется 1 из ранее пройденных навыков (30% объёма).
6. **Ошибка** → REWRITE (§6.1) + предложение возвращается в очередь через 2–4 позиции, пока не будет отвечено верно.
7. **Финал**: сводка (точность, время, ошибки по тегам) + при наличии ключа — 1–3 персональных предложения от LLM на самую частую ошибку сессии + кнопка «Разобрать с репетитором».
8. **Опционально**: голосовой раунд — 5 предложений из сессии устно (§9.4).

### 3.4 Карта навыков v1.1 (~76 навыков в ядре A1–B2 + 8 навыков C1 в бэклоге)

Список собран из English Profile (EGP/EVP), синлабусов British Council и Test-English, списка паттернов юзера. Это **нормативный минимум**; допускается коррекция ±10% на фазах Ф1/Ф6 с фиксацией в журнале. `id` — стабильные, не переименовывать.

**Решение 2026-07-07:** формальная книжная грамматика (инверсии, cleft-конструкции, номинализация — весь блок C1) даёт куда меньше ощущения разговорной беглости, чем высокочастотные разговорные связки/чанки (hedges, filler-фразы). Поэтому C1 **перенесён из ядра курса в бэклог** (см. правку в §16 Ф9) — не отменён, но не приоритет; ядро A1–B2 вместо него получает новый модуль разговорных связок (B1-5, ниже) — это то, что реально двигает ощущение уверенности в живой речи.

**Методика B1-5 (2026-07-08, итог ресёрча):** связки — не словарные единицы, а речевые инструменты; дриллим их как микросценарии, а не как «kind of = типа». Конкретно: (а) `theory_ru` навыка B1-5 описывает **функцию** фразы (что она делает: смягчает, уточняет, меняет тему), **позицию** в предложении и когда она звучит естественно, а когда чрезмерно/кринжово; (б) стимулы — гибрид: обычный перевод с живыми разговорными аналогами («типа», «ну», «вроде», «кстати», «честно говоря» — кальки запрещены, §8.6 GAPFILL) + функциональная подводка там, где прямого аналога нет («смягчи утверждение: …», «уточни мысль: …»); (в) саб-тип `mixed` для B1-5 = выбор между близкими связками (I mean / kind of / sort of / like) — та же механика, что контраст-дуэль §6.3; (г) в навыке 3–6 связок, добитых до автоматизма, а не 50 сразу; (д) Free Talk после прохождения B1-5 подмешивает пройденные связки в темы разговора — перенос в живую речь (§8.9).

### 4.2 Схема пайплайна

```
fetch → clean → pair → tag(rules) → tag(llm) → level → curate → gapfill → enrich → emit → validate → [audio]
```

Стек пайплайна: Python ≥3.12 (в системе 3.14), `uv` (есть), polars, spaCy `en_core_web_sm`, wordfreq, `google-genai`, typer (CLI `etr`), sqlite3 (кэш LLM-вызовов), pytest. Отдельная папка `pipeline/` в монорепо (§5.1). Все шаги идемпотентны и резюмируемы; промежуточные артефакты в `data/` (gitignored), результат — `packs/*.json` (коммитится). `fetch` шлёт браузерный User-Agent — manythings отдаёт 406 «голым» клиентам (замечено 2026-07-07).

CLI:

```bash
uv run etr fetch      # скачать rus-eng.zip (+ опц. полные экспорты Tatoeba) в data/raw/
uv run etr clean      # нормализация/фильтры → data/clean/pairs.parquet
uv run etr tag        # rule-based теггер + LLM-пасс для неоднозначных → data/tagged/
uv run etr level      # CEFR-оценка лексики (§4.5)
uv run etr curate     # квоты, диверсити, саб-дриллы, difficulty → data/curated/
uv run etr gapfill    # LLM-догенерация дефицитных навыков (§4.7)
uv run etr enrich     # theory_ru, common_errors, probe items, youglish_query
uv run etr emit       # → packs/{skill_id}.json + packs/index.json
uv run etr validate   # линт паков: схема, квоты, дубли, лексика уровня
uv run etr audio      # (опционально, Ф9) прегенерация озвучки edge-tts
```

### 4.3 Clean: правила фильтрации

Вход: пары `(en, ru, attribution)`. Отбрасываем/чиним:

1. Unicode-нормализация NFC; кавычки/апострофы → прямые; схлопнуть пробелы.
2. Длина EN: 2–18 токенов (для паков A1–A2 верхний предел 12).
3. EN содержит только латиницу+пунктуацию; RU — кириллицу (допускается латиница в именах).
4. Дубликаты: точные и «почти» (нормализованная форма без пунктуации/регистра).
5. Профанити-фильтр по стоп-списку (better-profanity или собственный список).
6. Телеграфные/обрывочные строки (нет глагола и >2 токенов; всё капсом; многоточия).
7. **Группировка для accepted answers**: `group by normalize(ru)` → все связанные EN-варианты сохраняются в `en_accepted` будущего item (главный `en_main` = самый короткий частотный вариант). При использовании полного графа Tatoeba брать только прямые связи (не переводы переводов). **Фильтр паттерна (2026-07-08):** при попадании item в пак навыка каждый кандидат `en_accepted` прогоняется через детектор этого навыка (§4.4) и через LLM ok-фильтр наравне с главной парой; варианты, не использующие дриллимый паттерн, в `en_accepted` не попадают — иначе tier 1 засчитывал бы как correct то, что judge оценил бы лишь как «acceptable, паттерн обойдён» (§8.6, правило 2). Каскад не должен противоречить сам себе.
8. **Tom/Mary-скос Tatoeba**: предложений с именами собственными ≤20% на навык. Опциональный шаг `--name-swap`: парная замена по словарю (`Tom↔Том` → случайное имя из 20 пар `Alex↔Алекс`…); соответствие проверяется по таблице падежных форм (`Том/Тома/Тому/Томом/Томе ↔ Алекс/Алекса/…` — русские имена склоняются регулярно); нет соответствия формы — нет замены, предложение остаётся как есть (деградация безопасная).

### 4.4 Tag: разметка паттернов (два пасса)

**Пасс 1 — rule-based (spaCy POS/deps + regex), покрывает ~80% A1–B1.** Каждому навыку — детектор с required/forbidden признаками. Примеры (полная карта — `pipeline/src/etr/rules.py`, golden-тесты обязательны):

| Навык | Детектор (эскиз) |
|---|---|
| `a1_do_questions` | `^(do)\b … \?$`, root=VB (base), нет wh-слова в начале |
| `a1_does_questions` | `^(does)\b … \?$` |
| `a1_there_is` | лемма-старт `there` + be |
| `a2_going_to` | `(am|is|are|was|were) going to VB` |
| `a2_pres_cont` | aux be + VBG, нет `going to`, нет будущих маркеров |
| `a2_past_simple_*` | root VBD, нет aux have/be |
| `a2_pres_perf_intro` | `(have|has) … VBN` + (`ever|never`/`?`) |
| `b1_passive` | be + VBN, subject≠agent (нет VBG рядом) |
| `b1_second_cond` | `if` + VBD в придаточном + `would VB` в главном |
| `b1_relative_clauses` | `who|which|that|where` как relativizer (dep=relcl) |
| `b2_causative` | `(have|get|had|got) + NP + VBN` |
| `b2_modal_deduction_past` | `(must|might|may|can't|could) have VBN` |
| `c1_inversion` | старт `Not only|Hardly|Never|Rarely|No sooner` + инверсия aux |

Предложение может честно относиться сразу к **нескольким неконфликтующим навыкам** (например, and инверсия+эмфатический do) — не форсим единственного победителя искусственно; специфичный детектор побеждает только при реальном конфликте признаков, иначе кандидат сохраняет все совпавшие теги, и `curate` (§4.6) решает по каждому навыку отдельно, использовать ли предложение (кап: одно предложение суммарно ≤2 навыка, чтобы не плодить дежавю).

**Пасс 2 — LLM-верификация.** Обязательно для (а) ambiguous/конфликтующих кандидатов, (б) навыков с низкой точностью правил (B2–C1). Плюс **проверка на естественность** (используем избыток кандидатов — 536k пар на ~5–8k нужных предложений, можно быть разборчивым): для навыков A1–B1, где после rule-based тегирования кандидатов заметно больше квоты (§3.2), LLM-пасс прогоняется по всем кандидатам, а не по случайным 5%, и `curate` предпочитает предложения с `ok=true` от `TAGGER_SYSTEM` (естественность уже часть его контракта, см. §8.6) более стилистически «выхолощенным»/шаблонным Tatoeba-фразам, если выбор есть. Батчи по 25 предложений, temperature 0, JSON-ответ. Кэш в `pipeline/cache.sqlite` с ключом `(model, prompt_hash, sentence_id)` — перезапуск ничего не пересчитывает. Оценка объёма при таком расширенном покрытии — на порядок больше батч-вызовов, чем в узком варианте (5%), но это разовый офлайн-прогон, растянутый на несколько дней free tier — бюджет это переживает. ⚠️VERIFY актуальные лимиты (§17.2).

Пайплайн не привязан к Gemini: TAGGER/GAPFILL работают с любой моделью с JSON-режимом (OpenRouter, Groq, локальная 70B) — план Б на случай недоступности конкретного провайдера в регионе мейнтейнера или смены его условий.

### 4.5 Level: CEFR-оценка предложения

Правило 95% лексического покрытия: предложение уровня L, если ≥95% его знаменательных слов (лемм) входят в лексику ≤L, иначе уровень поднимается или предложение отбрасывается для этого навыка.

Диапазоны лексики по умолчанию — через `wordfreq` Zipf-частоту леммы (открыто, MIT):

| CEFR | Zipf-порог (лемма) | ~соответствие |
|---|---|---|
| A1 | ≥ 4.9 | ~top-1000 |
| A2 | ≥ 4.5 | ~top-2000 |
| B1 | ≥ 4.0 | ~top-4500 |
| B2 | ≥ 3.5 | ~top-9000 |
| C1 | < 3.5 | остальное |

Служебные слова (стоп-лист) не учитываются. Если юзер положил `data/local/oxford_cefr.csv` (Oxford 3000/5000 с уровнями) — он используется как приоритетный якорь, wordfreq — как fallback. Итоговый уровень предложения = **max(уровень навыка, лексический уровень wordfreq, cefr из LLM-пасса §4.4, если он был)** — LLM-оценка ловит полисемию и идиоматику, которые частота леммы не видит («get around to» состоит из топ-частотных слов, но это B2). Пороги калибруются на Ф1: (а) против открытых sentence-level CEFR-датасетов как ground truth (подтв. 2026-07-08: **CEFR-SP** — ~17k предложений с экспертной разметкой, **ReadMe++ en** — 2 822 предложения из 20 доменов; удобный доступ — коллекция UniversalCEFR на Hugging Face: `cefr_sp_en`, `readme_en`; лицензии CC BY-NC-SA — ок для некоммерческого проекта; C2 отфильтровать; словные списки CEFR-J/EVP — как доп. признак «самое сложное слово в предложении»); (б) ручная выборка 30 предложений/уровень; значения фиксируются в журнале.

Честное замечание о правиле 95%: при нашей длине предложений (3–8 знаменательных слов) порог математически эквивалентен «ноль знаменательных слов вне уровня» — и это желаемое поведение (полностью прозрачный стимул); «95%» — формулировка на вырост для длинных B2-предложений. Помнить об этом при разборе недоборов квот.

### 4.6 Curate: отбор в навык

Для каждого навыка из кандидатов отбираем до квоты (§3.2):

1. Лексика проходит уровень (§4.5), длина в лимите уровня.
2. **Диверсити**: не более 4 предложений на одну глагольную лемму; не более 20% с именами собственными; сортировка на разнообразие тематики (грубая эвристика по ключевым существительным).
3. Распределение по саб-типам (affirm/neg/question/wh/short_answer/mixed) согласно §3.2.
4. `difficulty` 1–5 внутри навыка: по длине + лексическому уровню + наличию модификаторов.
5. Если после отбора < 60% квоты → для B1–C1 навык попадает в очередь `gapfill`; для A1–A2 сначала снижается квота до нижней границы (§3.2), gapfill — крайняя мера (конфликт двух правил «недобор → генерация» и «generated в A1–B1 → 0%» §4.7 решается в пользу меньшего, но корпусного навыка).

### 4.7 Gapfill: LLM-догенерация (офлайн)

Для дефицитных навыков (в основном B2–C1) Gemini генерирует пары по промпту `GAPFILL_SYSTEM` (§8.6): бытовые темы, лексика в пределах уровня, 4–12 слов, разные глаголы, `en_accepted` с 1–3 альтернативами. Затем **валидационный пасс моделью другого вендора** (`TAGGER_SYSTEM` с ok-флагом; не тем же семейством, что генерировало — систематические ошибки пары проходят парой) + опционально self-hosted LanguageTool (§13.3). Помечается `source: "generated:<model>:<date>"`. Доля generated в навыке ≤50%, в A1–B1 стремиться к 0%. **Исключение (2026-07-08): модуль B1-5 «Разговорные связки» — до 90% generated** (разговорные чанки редки в Tatoeba, а их генерация — самый безопасный вид: формулы фиксированы, лексика простая); в остальных навыках лимит жёсткий.

### 4.8 Формат пака (нормативная схема)

`packs/{skill_id}.json`:

```json
{
  "schema_version": 1,
  "skill": {
    "id": "a1_do_questions",
    "cefr": "A1",
    "module": "a1_m2",
    "module_title_ru": "Настоящее простое",
    "title_ru": "Вопросы с Do you…?",
    "pattern": "Do + you/we/they + V1 …?",
    "theory_ru": "3–6 предложений…",
    "common_errors": [
      {"tag": "aux_missing", "ru": "Пропуск do: ~You like tea?~ → Do you like tea?"}
    ],
    "probe_item_ids": ["s_00123", "s_00456", "s_00789"],
    "youglish_query": "do you usually"
  },
  "items": [
    {
      "id": "s_00123",
      "ru": "Ты часто забываешь имена?",
      "en_main": "Do you often forget names?",
      "en_accepted": ["Do you forget names often?"],
      "sub": "question",
      "difficulty": 2,
      "cefr_lex": "A1",
      "source": "tatoeba:1234567",
      "attribution": "CK"
    }
  ]
}
```

`packs/index.json`: версия курса, список уровней→модулей→навыков (id, title_ru, cefr, count, checksum файла) — по нему приложение строит карту курса и качает паки лениво.

Размер: пак ~20–40 КБ, весь курс A1–C1 < 4 МБ JSON — статика, кэшируется service worker'ом. `emit` детерминирован (стабильная сортировка навыков, items и ключей): повторный прогон без изменения данных даёт нулевой git-дифф; ревью регенерации — по сводке `validate` (сколько предложений добавлено/удалено по навыкам), а не по сырому диффу JSON.

### 4.9 Validate: линт паков (обязательный CI-шаг)

- JSON-схема (см. выше) валидна для каждого пака; все id уникальны глобально.
- Квоты и распределение саб-типов в допуске ±20%.
- Нет дублей нормализованных `ru` внутри навыка; `en_main ∈ en_accepted` запрещено (main хранится отдельно).
- Лексический уровень каждого item ≤ уровня навыка.
- `probe_item_ids` существуют; у каждого навыка есть theory_ru и ≥3 common_errors.
- Каждый `en_accepted` проходит детектор паттерна своего навыка (§4.3.7) — беспаттерновым вариантам в паке не место.
- Доля generated и доля имён собственных в лимитах.

## §6. Drill engine

### 6.1 Машина состояний предложения

```
show(ru) → input → check (каскад §7)
  ├─ correct/acceptable  → фидбек ✓ (+ «естественнее: …» если есть) → автоозвучка → next
  ├─ minor_error         → жёлтый фидбек: правка + тег → REWRITE → next (засчитано с пометкой)
  └─ wrong               → красный фидбек: краткое «что не так» → input остаётся:
       retry | hint (L1 формула → L2 пословное открытие: первые буквы → слово целиком) |
       give up | кнопки репетитора §8.5 («Ошибки» / «Разбор» / «Варианты» / «Нюансы»)
       → как только верный ответ стал видим юзеру (give up, «Ошибки», «Разбор»,
         corrected/natural от judge) — переход дальше ТОЛЬКО через REWRITE

  REWRITE (закрепление): верный ответ и разбор скрываются, поле очищается →
       юзер ВОСПРОИЗВОДИТ показанное предложение по памяти (не придумывает новый
       перевод — именно повторяет то, что только что видел) → проверка tier 1–2
       → успех → next; fail → ответ показывается снова → снова REWRITE (цикл до успеха)

  любой fail → предложение дополнительно возвращается в очередь через 2–4 позиции
       и попадает в разминку следующей сессии
```

**REWRITE — принципиальное отличие от English Dog**, где после разбора просто идёшь дальше: прочитать правильный ответ ≠ уметь его построить. Пока юзер не воспроизвёл предложение сам по пустому полю, «понял глазами» не превращается в навык (retrieval practice, §2). Цепочка: написал → ошибка → разбор + верный ответ → написал заново сам → (цикл до успеха) → дальше.

**Важное уточнение (иначе будет казаться багом):** REWRITE намеренно проверяется только tier 1–2 (точное/почти точное совпадение с ПОКАЗАННЫМ эталоном), а не полным каскадом с LLM-судьёй. Это не открытый перевод «ещё раз своими словами» — это retrieval только что увиденной модели. UI должен явно формулировать задачу как «повтори это же предложение», а не «переведи ещё раз» — иначе юзер попробует равноценный, но другой вариант и получит ложный fail.

**Эталоны REWRITE (2026-07-08):** в разборе юзеру могли быть показаны до трёх строк — en_main, corrected и natural от judge. REWRITE принимает ЛЮБУЮ из показанных (эталоны = все показанные строки); UI подсвечивает одну как основную. Идеальное воспроизведение natural не имеет права получить fail.

**Предохранитель от застревания:** после 2-го подряд фейла REWRITE эталон остаётся видимым на экране — перепечатка с опоры тоже закрепление, и цикл гарантированно завершается (иначе длинное B2-предложение — готовый rage-quit).

Фидбек любого неверного ответа обязан показывать **пословный диф** «твой вариант ↔ эталон» с подсветкой расхождений — явное замечание разрыва между своим выводом и образцом (noticing, §2) работает само по себе, ещё до объяснений.

Подсказки, retry и REWRITE не «наказываются» очками — наказание разрушает мотивацию; но предложение с fail попадает в разминку следующей сессии.

### 6.2 Построитель очереди сессии

Параметры: `newTarget` (12–20), `interleaveRatio` (0.3 — конвенциональная эвристика, настраиваемый параметр движка, не научная константа), `reviewWarmup` (5–8). Край «пройденных навыков ещё нет» (первый навык курса): interleave-вставки пропускаются — обязательный кейс табличных тестов. Алгоритм: разминка из due/failed → новые предложения навыка по difficulty 1→5 с саб-дриллами в порядке affirm → question → neg → wh → mixed; каждое ~3-е предложение — interleave из пройденных навыков модуля/уровня (равновероятно, приоритет недавним ошибкам). Requeue вставляется на позицию +2..4.

### 6.3 Типы сессий и пресеты

| Тип | Материал | Особенность |
|---|---|---|
| Дрилл навыка (§3.3) | новые предложения навыка | ядро |
| Review (§10) | due-навыки, свежие предложения | SRS |
| **Fluency-спринт (4/3/2)** | только освоенное (accuracy навыка >90%) | один и тот же набор ~10 предложений проходится 3 круга с уменьшающимся лимитом (4 → 3 → 2 мин, схема Nation 4/3/2); беглость строится на лёгком материале с темпом; ошибки не штрафуются |
| **Контраст-дуэль** | mixed-пары двух путающихся навыков | вперемешку без предупреждения: Past Simple vs Present Perfect, will vs going to, some vs any, gerund vs infinitive; статическая таблица «дуэлей» в коде; предлагается, когда оба навыка пройдены — учит ВЫБИРАТЬ форму, а не только строить её |
| Охота на ошибки (§10.4) | навыки топ-тегов юзера | персонализация |
| Listening / Speaking (§9) | пройденные предложения | смена модальности |
| **Свободный диалог** (§8.9) | нет заданных предложений — свои мысли | единственный режим без русского стимула; закрывает разрыв «дрилл ≠ разговор» (§17.1) |

Пресеты длительности при старте: **5 мин** (~10 предложений, «нет времени» — streak не сгорает), **15 мин** (стандарт), **25 мин** (полная). Дневной план (§10.5) чередует модальности по дням: письмо → голос → аудирование.

### 6.4 Управление

Enter — проверить; повторный Enter — дальше. `Ctrl+H` — подсказка, `Ctrl+G` — сдаться, `Ctrl+P` — озвучить. Автофокус инпута всегда. Инпут дрилла: `autocorrect=off, spellcheck=false, autocomplete=off, autocapitalize=off` (мобильные клавиатуры уважают это частично — принять: предиктив помогает со спеллингом, но порядок слов, aux и формы юзер всё равно строит сам). На мобиле — крупные кнопки под инпутом. У каждого предложения меню «⚑ плохое предложение» → локальный блэклист (исключается из ротации) + экспорт `blacklist.txt` из настроек (скармливается пайплайну при следующей сборке паков).

## §7. Проверка ответа (каскад)

Главная проблема продукта: «перевёл правильно, но не так, как ожидалось». Решение — каскад, где LLM только на верхнем этаже, с кэшем.

### 7.1 Нормализация (обе стороны: ввод и эталоны)

`normalize(s) → Set<string>` (множество канонических форм из-за ветвлений неоднозначности):

1. NFC, lowercase, типографские апострофы/кавычки → прямые, схлопнуть пробелы, убрать финальные `.!?`.
2. Контракции → полные формы по таблице (`don't→do not`, `won't→will not`, `can't→cannot`, `i'm→i am`, `they're→they are`, `let's→let us`, …). Неоднозначные `'s`/`'d` дают ветвление: `he's → {he is, he has}`, `i'd → {i would, i had}` — результат = множество вариантов.
3. Числительные-слова 0–100 → цифры (`two → 2`).
4. BrE → AmE по явному словарю ~40 слов (`colour→color`, `favourite→favorite`, `centre→center`, `organise→organize`, `travelling→traveling`, `grey→gray`, `programme→program`, `neighbour→neighbor`, `realise→realize`, `theatre→theater`, …) — файл `app/src/checker/britishAmerican.ts`. Туда же — конечный список морфологических пар BrE↔AmE (~10 шт.): `learnt↔learned`, `spelt↔spelled`, `burnt↔burned`, `dreamt↔dreamed`, `spoilt↔spoiled`, `leant↔leaned`, `gotten↔got`. Никаких суффиксных эвристик (ложные срабатывания типа *hour*).

### 7.2 Каскад

```
tier 0  PRE-CHECK  локально, до каскада: пустой ввод / кириллица / вставленный RU-стимул целиком
                   → мгновенный wrong «ответ должен быть по-английски», LLM-вызов не тратится
tier 1  EXACT      normalize(user) ∩ normalize(en_main ∪ en_accepted ∪ acceptedCache[ruHash]) ≠ ∅  → correct
tier 2  NEAR       наилучший эталон по token-diff; допускается ≤1 опечатка на предложение:
                   слово len≥4, edit distance 1, И правка НЕ создаёт/убирает морфемный суффикс
                   {s, es, ed, ing, er, est}, не меняет служебное слово (a/an/the/do/did/is…),
                   И слово юзера НЕ является само по себе словарным словом (компактный список
                   top-10k английских форм, генерится из wordfreq в пайплайне, ~50 КБ):
                   want/went, than/then, lose/loose — не опечатки, а ДРУГИЕ слова → дальше по каскаду.
                   → correct с пометкой «опечатка: …» (+ тег spelling в attempts/errorProfile,
                   иначе статистика опечаток видит только judge-случаи).
                   Разница в артикле/предлоге/форме глагола — НЕ опечатка.
tier 2.5 (опция)   self-hosted LanguageTool, если настроен (§13.3): чистая грамматика юзера +
                   совпадение мешка лемм с эталоном → кандидат в acceptable, идёт на tier 3 за подтверждением
tier 3  LLM JUDGE  если есть ключ и бюджет: промпт §8.6-JUDGE, JSON, zod, temperature 0,
                   timeout 8s (обычная латентность 1–2 с, 8 с — потолок), 1 retry.
                   UI ожидания: «спрашиваю ИИ…» + кнопка «Не ждать — оценю сам» (мгновенный tier 4).
                   Повторный parse-fail → tier 4 с тостом «ИИ-ответ не разобрался — проверь себя»;
                   при активном кастомном промпте — подсказка «ты правил промпт judge, попробуй сбросить».
                   verdict=correct|acceptable и add_to_accepted → записать в acceptedCache
tier 4  SELF       нет ключа/бюджета/сети: показать эталоны + кнопки «Я был прав» / «Ошибся»
                   (verdictSource='self'; честность — забота юзера, это его обучение)
```

Экономика: в день-1 до LLM доходит ~30% попыток, к неделе — <10% (кэш растёт). При дневной цели 30–60 предложений это 5–20 вызовов/день — на порядки меньше free-tier лимитов (§17.2).

### 7.3 Теги ошибок (нормативный enum)

```
word_order | verb_tense | verb_form | aux_missing | article | preposition |
agreement | pronoun | vocab_choice | spelling | word_missing | word_extra | unnatural
```

Judge обязан использовать только их (zod-enum). По ним строится errorProfile и режим «охота на ошибки» (§10.4).

### 7.4 Контракт judge (zod)

```ts
const JudgeVerdict = z.object({
  verdict: z.enum(["correct", "acceptable", "minor_error", "wrong"]),
  error_tags: z.array(z.enum(ERROR_TAGS)).max(3),
  explanation_ru: z.string().max(280),   // ≤ 2 коротких предложения
  corrected: z.string(),                 // минимальная правка ответа юзера
  natural: z.string(),                   // как сказал бы носитель
  add_to_accepted: z.boolean(),
});
```

Пост-валидация языка: при {UI_LANG}=ru `explanation_ru` обязана содержать кириллицу (дешёвые модели стабильно путают язык ответа при английском системном промпте) — иначе один retry с усиленной инструкцией, потом tier 4; кейс — в фикстуры §15. LEVEL в промпте judge = уровень НАВЫКА (строгость привязана к паттерну); tutor и Free Talk получают уровень ЮЗЕРА — они расходятся при interleave/review (§8.6).

### 7.5 Контроль качества вердиктов (LLM тоже ошибается)

Дешёвая модель на роли judge — не оракул: может неправильно отклонить валидный вариант или неправильно принять ошибку. Раз это единственная обратная связь, на которой строится вся идея автоматизма, нужен минимальный контроль качества:

- На каждом вердикте — маленькая кнопка **«🚩 Мне кажется, ИИ ошибся»**. Клик логирует запись (item, ответ юзера, вердикт, модель, timestamp) в локальную таблицу `judgeDisputes` (Dexie) — без отправки куда-либо. На вердикте correct/acceptable флаг дополнительно предлагает **убрать этот вариант из acceptedCache** — ошибочный add_to_accepted не должен жить вечно (иначе tier 1 навсегда засчитывает неверный ответ); в «Настройки → Данные» — кнопка «очистить кэш принятых ответов».
- Экран «Настройки → Данные» показывает счётчик и позволяет экспортировать `judgeDisputes` в JSON — для собственной проверки или ручной перепроверки другой моделью.
- Явная рекомендация в README: если юзер сомневается в вердикте — переключить модель на другую (§8.8) и переспросить то же предложение; расхождение между двумя независимыми моделями — сильный сигнал, что вопрос спорный, а не однозначная ошибка юзера.
- **Мини-бенчмарк судьи (Ф3а, до выбора дефолтной модели):** 60–100 вручную размеченных кейсов «ответ юзера → ожидаемый вердикт» (артикли, предлоги, «паттерн обойдён», BrE/AmE, minor vs wrong) — эти же кейсы служат фикстурами vitest LLM-контрактов (§15). Прогнать через модели-кандидаты, точность зафиксировать в журнале; модель с совпадением <80% в дефолтный роутинг judge не ставится (фолбэк на неё заменяется честным tier 4 — тумблер §8.1). Без бенчмарка вся пирамида «автоматизм строится на обратной связи» стоит на непроверенном судье.

## §8. AI-слой (BYOK)

### 8.1 Интерфейс провайдера

```ts
type Role = "judge" | "tutor" | "generator";
interface ChatRequest { system: string; messages: Msg[]; json?: boolean; maxTokens?: number; temperature?: number } // judge и все JSON-роли зовутся с temperature 0 — иначе вердикты недетерминированы
interface LLMProvider {
  id: string; label: string;
  isConfigured(): boolean;                       // ключ введён / URL доступен
  chat(req: ChatRequest, signal?: AbortSignal): Promise<string>;
}
```

Роутинг ролей — конфиг в настройках (редактируемый список фолбэков):

```json
{
  "judge":     ["gemini:flash-lite", "groq:llama-8b", "ollama:default"],
  "tutor":     ["gemini:flash", "openrouter:free", "ollama:default"],
```

Алиасы моделей — в одном файле конфига (на 2026-07: `flash` = gemini-3.5-flash, `flash-lite` = gemini-3.1-flash-lite, `groq:llama-8b` = llama-3.1-8b-instant); обновляются без правки кода. Рядом с редактором роутинга — кнопка «Сбросить к дефолту» и тумблер «при недоступности основного судьи — сразу самопроверка» (не фолбэчить на слабую модель; кому слабый судья хуже никакого — см. бенчмарк §7.5).

README честно объясняет: «недоступен один провайдер — выбери другого из списка» (§18).

**Валидация ключа при вводе:** сохранение облачного ключа сопровождается тестовым мини-вызовом («Проверить» — как у локальных профилей §8.7); юзер не должен узнавать о невалидном ключе посреди первой сессии.

### 8.3 Бюджеты

Дневные счётчики per provider+role в `providerBudget`. Дефолты: judge 800/д, tutor 150/д, generator 50/д (правится в настройках) — это потолки-предохранители, не смета: реальный расход judge 5–20/д (§7.2). Зачем локальные лимиты, если провайдер сам шлёт 429: (а) защита платного ключа — там 429 не придёт, придёт счёт; (б) распределение дня по ролям — иначе Free Talk может съесть бюджет судьи; (в) предсказуемость UX. Локальный счётчик не видит расход с других устройств с тем же ключом — истинный лимит энфорсит провайдер (429 → автопереключение §8.8), счётчик — превентивная вежливость. `providerBudget.date` считается по часовому поясу провайдера (Gemini — Pacific); поповер чипа показывает «сбросится через N ч» в локальном времени юзера. Исчерпан → тихий даунгрейд на следующий фолбэк → tier 4; баннер объясняет причину человеческим языком: «дневной лимит бережёт бесплатный тариф твоего ключа — завтра продолжим с ИИ». Никогда не блокировать обучение из-за LLM.

### 8.4 AI-урок по запросу

Экран: юзер пишет тему свободно («телефонные разговоры», «прошедшее время про путешествия») → `generator` строит урок из 15 предложений (промпт LESSON_GEN) → урок проходит через обычный drill engine. Предложения эфемерны (не в SRS), но зафейленные становятся пиявками (§10.3), а сам урок можно сохранить кнопкой (уходит в Dexie как локальный пак `custom_*`). Это «ИИ генерирует целый урок за один запрос» — дёшево: 1 вызов = целая сессия.

Разновидность — **«Мои слова»** (аналог «Мой словарь» в English Dog): юзер вводит список английских слов → приложение (а) находит предложения с этими словами в скачанных паках и (б) при наличии ключа догенеривает дрилл-предложения с ними через generator. Слова тренируются только внутри предложений — отдельных карточек «слово→перевод» не делаем (§1.3).

Ещё разновидность — **«Про себя»** (self-reference/personalization effect): после закрытия навыка приложение предлагает написать 2–3 **своих** предложения с этим паттерном о своей жизни («Расскажи о себе через Present Perfect»). Judge проверяет и правит; принятые предложения сохраняются как личные примеры навыка и вплетаются в будущие повторения — материал о себе запоминается сильнее чужих Томов и Мэри, и это первый шаг от дрилла к свободной речи. Экран напоминает мелкой строкой: написанное уходит твоему LLM-провайдеру (общий принцип приватности ИИ-режимов, §14.3).

### 8.5 Репетитор: 4 действия + чат

Два слоя:

1. **Действия-кнопки** (как у English Dog): «Ошибки» / «Разбор» / «Варианты» / «Нюансы». Каждая — один одноразовый LLM-вызов со своим промптом (§8.6, блок ACTION_*) на полном контексте: RU, ответ юзера, верный ответ + accepted-варианты, verdict, error_tags, паттерн, уровень. **Кэш**: результаты «Вариантов» и «Нюансов» зависят только от предложения → кэшируются в Dexie per item навсегда (повторные клики и повторные встречи предложения бесплатны); «Ошибки» и «Разбор» зависят от ответа юзера → кэш per (item + hash нормализованного ответа). Ключ кэша всех действий дополнительно включает hash активного промпта действия — правка промпта в настройках автоматически инвалидирует старые кэшированные ответы (иначе юзер видит ответы на старый промпт как будто на новый).
2. **Чат** («Спросить репетитора»): свободные до-вопросы после действий, сократический стиль (TUTOR_SYSTEM), максимум 6 ходов, потом мягкое «вернёмся к дриллу». История чата не персистится (экономия и простота); в errorProfile уже лежат теги исходной ошибки из judge-вердикта — из самого чата ничего дополнительно не извлекается (и не нужно; прежняя формулировка «выжимка из чата» описывала несуществующий механизм).

Все вызовы — из бюджета tutor (§8.3). Любое действие, раскрывшее верный ответ («Ошибки», «Разбор», give up), включает обязательный REWRITE (§6.1).

**Редактор промптов (Настройки → Модели ИИ):** каждый рантайм-промпт из §8.6 (JUDGE_SYSTEM, TUTOR_SYSTEM, ACTION_ERRORS/EXPLAIN/VARIANTS/NUANCES, FREETALK_SYSTEM, FREETALK_SUMMARY_SYSTEM, LESSON_GEN_SYSTEM) показывается как обычный текст в текстовом поле — можно править и сохранить, кнопка «Сбросить к дефолту» рядом с каждым. Правки хранятся локально (Dexie `kv`, per-промпт) и подставляются вместо дефолта при следующем вызове этой роли/действия. Промпты пайплайна (TAGGER_SYSTEM, THEORY_GEN_SYSTEM, GAPFILL_SYSTEM) не редактируются в приложении — они офлайн и в рантайм не попадают. При правке JUDGE_SYSTEM — предупреждение прямо в редакторе: ответ обязан остаться валидным JSON по схеме §7.4. Сохранение кастомного промпта JSON-роли запускает **smoke-тест**: один тестовый вызов + zod-валидация → «промпт рабочий» / «ответ не парсится — исправь или сбрось» (тихий отказ главной функции из-за правки текстового поля недопустим). Дефолтные промпты версионируются (hash): если апдейт приложения изменил дефолт, а у юзера сохранён кастом — пометка в редакторе «дефолт обновился».

### 8.6 Промпты (нормативные, английский)

UI двуязычный (§12) → все инструкции «in Russian» и поля `*_ru` параметризуются языком интерфейса `{UI_LANG}` (ru|en): при английском UI объяснения пишутся по-английски. В промптах ниже «Russian» читать как «{UI_LANG}»; имена полей в схемах не меняются.

**JUDGE_SYSTEM** (рантайм, role=judge):

```
You are a strict but fair grader of English translations for Russian-speaking learners.
Input: RU (stimulus), USER (learner's answer), REFS (known correct translations),
PATTERN (the grammar pattern being drilled), LEVEL (CEFR).
Rules:
1. USER need not match REFS word-for-word. Any natural, grammatically correct English
   sentence with the same meaning that USES the drilled PATTERN is correct/acceptable.
2. If USER is correct but avoids the drilled PATTERN (paraphrase), verdict="acceptable",
   add_to_accepted=false, explanation_ru must say the pattern was avoided.
3. Ignore capitalization, terminal punctuation, contractions, US/UK spelling.
4. minor_error = meaning preserved, exactly one small slip (article, preposition, single
   wrong form). wrong = meaning changed, pattern broken, or 2+ grammar errors.
5. explanation_ru: Russian, max 2 short sentences, about THE error only. No praise, no lecture.
6. corrected = USER's sentence with minimal edits. natural = how a native speaker would say it.
7. error_tags only from: [word_order, verb_tense, verb_form, aux_missing, article, preposition,
   agreement, pronoun, vocab_choice, spelling, word_missing, word_extra, unnatural].
8. add_to_accepted=true ONLY if USER is fully correct AND uses the drilled pattern.
Respond with ONLY valid JSON:
{"verdict":"correct|acceptable|minor_error|wrong","error_tags":[],"explanation_ru":"",
 "corrected":"","natural":"","add_to_accepted":false}
```

**TUTOR_SYSTEM** (рантайм, role=tutor):

```
You are a warm, concise English tutor for a Russian-speaking learner (CEFR {LEVEL}).
Context: drilling pattern "{PATTERN}"; the learner made this mistake:
RU="{RU}", their answer="{USER}", correct="{REF}", error_tags={TAGS}.
Rules:
- Reply in Russian; English only inside examples. Max 120 words per reply.
- Socratic first: if the learner can plausibly find the fix, ask ONE guiding question
  instead of explaining. Explain directly on the second miss.
- One concept per reply. Plain words, no linguistics jargon.
- When the learner gets it, give exactly 2 fresh RU→EN micro-examples of the same pattern.
- Never drift off the current mistake. Never grade — grading happens elsewhere.
```

**Действия-кнопки репетитора** (рантайм, role=tutor; §8.5). В начало каждого из четырёх промптов подставляется общий контекст:

```
CONTEXT
RU (task): "{RU}"
LEARNER (their answer): "{USER}"
CORRECT (main): "{REF}"; also accepted: {REFS}
VERDICT: {VERDICT}; error_tags: {TAGS}
PATTERN drilled: "{PATTERN}" (CEFR {LEVEL})
Write in Russian; English only inside examples. Plain language, no linguistics jargon.
```

**ACTION_ERRORS** («Ошибки»):

```
List the learner's mistakes, one by one. For each: quote the wrong fragment → show the fix
→ explain WHY in one short sentence (rule or usage).
Max 3 mistakes, most important first; if more remain, add one line "Ещё мелочи: …".
If the answer is fully correct — say so and name one thing done well.
Format: numbered list. Under 90 words total.
```

**ACTION_EXPLAIN** («Разбор»):

```
Explain how the CORRECT sentence is built, like a tutor at a whiteboard:
1) skeleton: the word order / pattern formula applied to THIS sentence;
2) why the key forms are used (tense, auxiliary, article) — only those that matter here;
3) one memory hook: a mini-rule or analogy the learner can reuse in similar sentences.
Do not analyse the learner's answer here (that is the Errors button). Under 110 words.
```

**ACTION_VARIANTS** («Варианты»; кэш per item — ответ юзера в контекст не подставляется):

```
Give 3-5 natural English translations of RU, from most standard to conversational.
Label each: (нейтральный) / (разговорный) / (формальный) where relevant.
After each — max one short line in Russian: when this version fits.
Only sentences a native speaker would actually say; no invented rare phrasings.
```

**ACTION_NUANCES** («Нюансы»; кэш per item — ответ юзера в контекст не подставляется):

```
Point out the subtleties of this sentence that Russian speakers usually miss:
close synonyms and which one natives prefer here, articles, prepositions, collocations,
register, false friends. Max 4 bullets, one sentence each. Only nuances REAL for this
sentence — no generic grammar lecture. If there is a classic RU-speaker trap, start with it.
```

**FREETALK_SYSTEM** (рантайм, role=tutor; §8.9):

```
You are a friendly English conversation partner chatting with a Russian-speaking learner
(CEFR {LEVEL}) about: "{TOPIC}".
Rules:
- Speak ONLY natural, everyday spoken English. Never switch to Russian, never explain
  grammar mid-conversation, never grade.
- Keep each reply short: 1-3 sentences, almost always with a genuine follow-up question —
  this is a real back-and-forth, not a lecture.
- Match vocabulary and complexity to LEVEL; sound like a real person, not a textbook.
- Minor mistake that does not block meaning → do NOT correct directly. Recast it: reuse
  the corrected form naturally in your own next sentence, in passing.
- Mistake that blocks meaning → ask a brief clarifying question instead of explaining.
- Stay warm and curious; never break character as a conversation partner.
- If the learner writes in Russian or freezes, gently encourage them in English and offer
  one simple phrase to continue with, then keep the conversation going.
- If asked what a word or phrase means, explain briefly in simple English and move on —
  that is a normal conversation move, not a grammar lecture.
```

**FREETALK_SUMMARY_SYSTEM** (рантайм, role=tutor; §8.9, вызывается один раз при завершении диалога):

```
Given this English conversation transcript between you (tutor) and a Russian-speaking
learner (CEFR {LEVEL}), write a summary for the learner in Russian: 2-3 RECURRING mistake
patterns (not one-off typos), each with one corrected example taken from the learner's own
messages, phrased kindly. If nothing recurs, name 1-2 specific things expressed well instead.
Output ONLY valid JSON: {"summary_ru":"","recurring_tags":[]}
recurring_tags: only from the standard error enum where it clearly applies, else omit.
Under 120 words in summary_ru.
```

**LESSON_GEN_SYSTEM** (рантайм, role=generator):

```
You create a one-shot drill lesson for a Russian-speaking English learner.
Input: TOPIC (free text), LEVEL (CEFR), N (default 15), WEAK_TAGS (learner's frequent errors).
Output ONLY valid JSON:
{"title_ru":"","theory_ru":"3-5 short sentences","items":[
  {"ru":"","en_main":"","en_accepted":[""],"sub":"affirm|neg|question|wh|mixed"}]}
Rules: natural everyday sentences, 4-12 words; vocabulary within LEVEL; idiomatic Russian
stimuli (never word-by-word calques); vary verbs and subjects; no rare proper nouns;
bias 30% of items toward WEAK_TAGS if given; en_accepted = 1-3 alternative correct translations.
```

**TAGGER_SYSTEM** (пайплайн):

```
You classify English sentences for a grammar-drill course.
Input: SKILLS = [{id, pattern, description}]; SENTENCES = [{id, en, ru}].
For each sentence decide: (a) skills = up to 2 skill ids it clearly drills (empty array
if none fit well — do not force a weak match); (b) cefr = A1..C1 considering both grammar
and vocabulary; (c) ok=false if awkward, unnatural-sounding, offensive, archaic,
telegraphic, or not self-contained (this is also the naturalness filter, be strict here);
(d) ru_ok=false if the Russian translation is unnatural or does not match the English
meaning; watch for literal calques from English (e.g. "Я имею машину" for "I have a car").
Be conservative: only include a skill if the sentence is a genuinely good,
natural example of it — prefer an empty list over a forced weak match.
Output ONLY valid JSON: {"results":[{"id":"","skills":[],"cefr":"","ok":true,"ru_ok":true}]}
```

**THEORY_GEN_SYSTEM** (пайплайн):

```
Write a micro-theory block in Russian for skill "{TITLE}" (CEFR {LEVEL}, pattern: {PATTERN}).
Output JSON: {"theory_ru":"","common_errors":[{"tag":"","ru":""}],"youglish_query":""}
theory_ru: 3-6 short sentences, plain language, one formula line, 2 inline examples,
no linguistics jargon. common_errors: 3-5 typical mistakes of Russian speakers with this
pattern (tag from the standard enum; ru = one line with a wrong→right mini example).
youglish_query: 2-4 English words capturing the pattern for a YouGlish search.
```

**GAPFILL_SYSTEM** (пайплайн):

```
You write drill sentences for skill "{TITLE}" (pattern: {PATTERN}, CEFR {LEVEL}).
Generate {N} pairs. Output ONLY valid JSON:
{"items":[{"ru":"","en_main":"","en_accepted":[""],"sub":""}]}
Constraints: everyday topics (home, work, food, travel, plans, feelings, small talk);
vocabulary strictly within CEFR {LEVEL}; 4-12 words; no verb lemma used twice; natural
idiomatic Russian; en_accepted = 1-3 alternatives; follow this sub-type mix: {MIX}.
```

### 8.7 Локальные модели: Ollama / LM Studio / llama.cpp (доп. возможность)

Все три сервера отдают **OpenAI-совместимый API** (`/v1/chat/completions`) → в приложении один универсальный адаптер `local-openai`. Юзер запускает сервер сам, в приложении прописывает только Base URL и модель.

| Сервер | Запуск | Base URL по умолчанию | CORS для браузера |
|---|---|---|---|
| Ollama | `ollama pull <model>` → `ollama serve` | `http://localhost:11434/v1` | `OLLAMA_ORIGINS="https://<домен-приложения>"` перед запуском |
| LM Studio | Developer → Local Server → Start | `http://localhost:1234/v1` | тумблер CORS в настройках сервера |
| llama.cpp | `llama-server -m model.gguf` | `http://localhost:8080/v1` | открыт по умолчанию; `--api-key` по желанию |

Экран «Настройки → Локальные модели»: несколько профилей, у каждого — Base URL, модель (кнопка «получить список» через `GET /v1/models`), API-ключ (если задан), temperature, max_tokens, таймаут; кнопка «Проверить соединение» (ping + тестовый мини-вызов). `localhost` с https-сайта работает (Chrome считает localhost безопасным контекстом); доступ по IP локальной сети (`http://192.168.x.x:11434/v1`) может упереться в mixed content — обход описать в README (§18). Точные команды включения CORS ⚠️VERIFY на Ф3.

Локальные профили участвуют в роутинге ролей (§8.1) наравне с облачными — обычно как последний фолбэк перед tier 4.

### 8.8 Индикатор и переключатель активной модели (шапка приложения)

- В шапке всегда виден чип **«⚡ gemini-3.5-flash»** — модель, которая будет использована следующим LLM-вызовом (= первый настроенный и не исчерпавший бюджет пункт роутинга).
- Тап по чипу → поповер: все настроенные модели со статусом (✅ доступна / 🔒 дневной лимит / ⚠️ нет ключа / 🔌 сервер недоступен) и остатком бюджета (§8.3); ссылка «Настройки моделей».
- Выбор модели в поповере = **ручной override** для всех ролей — до конца дня или до нажатия «вернуть авто».
- **Автопереключение**: при 429/quota-ошибке провайдера или исчерпании локального счётчика вызов уходит следующему в цепочке; чип обновляется, тост «Gemini: лимит на сегодня → переключился на Groq». Обучение никогда не останавливается из-за лимита (в худшем случае — tier 4, чип показывает «локальный режим»).
- Каждый ответ ИИ подписан моделью мелким шрифтом; в `attempts` пишется поле `model` — в статистике видно, кто как судил.

### 8.9 Свободный диалог (Free Talk)

Прямой ответ на главный концептуальный изъян плана (§17.1): дриллы тренируют перевод ЗАДАННОЙ мысли, а не порождение своей и не ведение разговора. Free Talk — единственный режим без русского стимула.

**Механика**: юзер выбирает тему (пресет-список: путешествия, работа, увлечения, планы на день, спорный вопрос… + своя тема текстом) → чат с `tutor`-ролью, где ИИ ведёт себя как собеседник, а не репетитор: короткие реплики (1–3 предложения), обязательный встречный вопрос, живой разговорный регистр под уровень юзера. **Коррекция мягкая и без остановки диалога**: мелкая ошибка не прерывает поток — ИИ незаметно повторяет исправленную форму в своей следующей реплике (recast), не читает лекцию; ошибка, ломающая понимание — короткий уточняющий вопрос вместо объяснения правила. По завершении диалога (кнопка «Закончить») — отдельный вызов даёт сводку: 2–3 повторяющихся паттерна ошибок по-русски с примером из собственных реплик юзера, `recurring_tags` уходят в `errorProfile` (§10.4) наравне с обычными дриллами.

**Голос — опционально**: как только в приложении есть STT/TTS (Ф5/Ф7), в Free Talk появляется тумблер «говорить голосом» — тот же чат, но ввод через STT и озвучка реплик ИИ; текстовый режим остаётся дефолтом (дешевле, не требует микрофона).

**Бюджет**: расходует роль `tutor` (§8.3); многоходовый диалог тратит бюджет быстрее одной кнопки-действия — лимит на диалог: 15 реплик юзера или исчерпание дневного бюджета tutor, что раньше; за 2–3 реплики до лимита — мягкое предупреждение («скоро закругляемся»), чтобы финал не был обрывом; при исчерпании — мягкое сообщение «бюджет ИИ на сегодня закончился, увидимся завтра» и переход к обычным дриллам, прогресс диалога не теряется. Требует настроенной модели (облачной или локальной, §8.7) — без неё режим недоступен, кнопка на экране «Сегодня» показывает подсказку «нужен ключ ИИ».

**Уровень и живучесть (2026-07-08):** режим комфортен с A2+ — на входе подсказка для A1; пресеты тем размечены по уровням («спорный вопрос» не предлагается на A2). Транскрипт активного диалога хранится в Dexie до явного завершения: закрыл вкладку — при следующем входе предложение «завершить вчерашний разговор?» → сводка и recurring_tags не теряются (иначе весь выхлоп диалога зависит от нажатия кнопки «Закончить»).

## §9. Аудио: TTS, STT, listening, speaking

### 9.1 TTS (цепочка, от лучшего к запасному)

1. **kokoro-js** (основной): Apache-2.0, модель 82M, квантованная ~86 МБ, работает в браузере (WASM/WebGPU, transformers.js/ONNX), 11+ английских голосов US/UK, мужские/женские (подтв. 2026-07, §17.2). Ленивая загрузка: кнопка «Включить качественный голос» → скачивание модели с прогрессом → кэш в CacheStorage → офлайн навсегда. Названия голосов (`af_*`/`am_*` — US женск./мужск., `bf_*`/`bm_*` — UK) уточнить по model card на Ф5.
2. **Web Speech `speechSynthesis`** (мгновенный fallback): системные голоса, работает везде, качество зависит от ОС.
   - Лёгкая альтернатива Kokoro для слабых устройств: **KittenTTS** (15–80 МБ, Apache-2.0, статус developer preview) или Piper/sherpa-onnx WASM — решить на Ф5 по факту производительности, необязательно.
3. **Gemini TTS** (BYOK-опция, не обязателен).
4. **edge-tts прегенерация** (пайплайн, Ф9, опционально): библиотека жива (v7.2.8, 2026-03), голоса en-US-Jenny/Guy, en-GB-Sonia/Ryan; прегенерить только топ-предложения A1–A2 в opus, если рантайм-TTS окажется недостаточен. Неофициальный API — только офлайн-прегенерация, никогда в рантайме.

Настройки: акцент US/UK, пол, скорость 0.7/0.85/1.0, автоозвучка правильного ответа (default on). После ~20 озвучек системным голосом — разовый ненавязчивый промпт «есть голос получше (86 МБ, офлайн навсегда)» — иначе про кнопку Kokoro никто не узнает.

### 9.2 Listening (2 режима)

- **Диктант**: слышу EN → печатаю EN. Проверка каскадом tier 1–2 + подсветка дифа по словам. До 3 повторов, регулировка скорости, без штрафов.
- **Понимание**: слышу EN → печатаю смысл по-русски. Проверка мягкая: нормализованное сравнение с `ru` + self-check кнопки (LLM не нужен).
- **Восстановление** (dictogloss): предложение играется **один раз** на нормальной скорости → пишу по памяти. Проверка: tier 1–2 против прослушанного эталона, не совпало — самооценка по показанному эталону (LLM не зовём); «смысловая близость важнее дословности» означает «self-check решает юзер» — семантической метрики у каскада нет и не обещано. Тренирует удержание чанков в рабочей памяти — мостик к реальному аудированию.

Материал — те же паки (предложения пройденных навыков), отдельная сессия с экрана «Сегодня».

### 9.3 STT

- **Web Speech `SpeechRecognition`** — порядок попыток (Chrome 139+ умеет on-device, подтв. 2026-07-07):
  1) локально: `SpeechRecognition.available()` → при необходимости `install()` языкового пакета → `recognition.processLocally = true` (приватно, офлайн);
  2) обычный облачный Web Speech (Chrome/Edge, нужна сеть);
  3) без STT — shadowing с самооценкой (услышал → повторил вслух → сам отметил).
  Язык en-US/en-GB по настройке акцента (для распознавания русского акцента разница пренебрежима — настройка в первую очередь про то, что юзер слушает).
  **Приватность и согласие (2026-07-08):** перед первым включением микрофона — один диалог: «облачное распознавание может отправлять голос в облако браузера (Google); on-device доступен в Chrome 139+; всегда можно печатать» — без этого продукт «без телеметрии» молча слал бы голос третьей стороне. Отказ в разрешении микрофона → короткая инструкция + текстовые режимы (speaking опционален, ничего не блокируется). Пункт «что уходит наружу» — на странице приватности (§14.3/§14.4).
- BYOK-апгрейды: MediaRecorder → **Groq Whisper** через `proxy/` (free 2000 RPD) или аудио в Gemini → транскрипт + краткая оценка произношения.
- Тяжёлый офлайн-вариант (Whisper WASM / Transformers.js) — бэклог, в основные фазы не входит.

### 9.4 Speaking (не основной режим; юзер выбирает голос или текст)

- **Перевод вслух**: RU на экране → говорю EN → STT-транскрипт → тот же каскад проверки → диф. **STT-ошибка ≠ ошибка юзера:** транскрипт всегда показан («я услышал: …»), рядом кнопка «Я сказал иначе» — попытка не засчитывается как fail; голосовые фейлы не порождают пиявок (§10.3) — распознавание недостаточно надёжно, чтобы наказывать за него.
- **Shadowing**: слышу эталон (TTS) → повторяю → STT-диф подсвечивает расхождения по словам.
- **Запись и сравнение**: MediaRecorder пишет мой голос → свой дубль и эталон TTS проигрываются встык → самооценка. Слышать себя рядом с эталоном — самый дешёвый тренажёр произношения, работает вообще без AI и сети.
- **Вслух после верного** (toggle, default on): после ✓ приложение предлагает произнести предложение вслух — без проверки, просто привычка. Production effect: произнесённое запоминается на 10–20% лучше (§2).
- Голосовой раунд в конце обычного дрилла (§3.3) — те же предложения, что уже написал: письмо и речь в один день закрепляют паттерн в двух модальностях.

## §10. Интервальные повторения (FSRS)

### 10.1 Карточки на НАВЫК, не на предложение

Ключевое решение: FSRS-карточка = навык. Повторение навыка тянет 6–10 **свежих или давно не виданных** предложений этого навыка — тренируется перенос паттерна, а не заучивание конкретной фразы. `ts-fsrs` v5, desired retention 0.9, дефолтные параметры FSRS-6.

### 10.2 Оценка сессии → рейтинг FSRS

| Точность сессии по навыку | Rating |
|---|---|
| < 60% | Again |
| 60–79% | Hard |
| 80–92% | Good |
| > 92% | Easy |

### 10.3 Пиявки (leeches)

Предложение, зафейленное ≥2 раз в разных сессиях → личная FSRS-карточка (оценка по факту ответа: fail=Again, с подсказкой=Hard, чисто=Good). Пиявки показываются в разминке и review-сессиях, пока не выйдут в стабильность >30 дней. Голосовые (STT) фейлы пиявок не создают (§9.4) — ненадёжность распознавания не должна попадать в SRS.

### 10.4 Охота на ошибки

`errorProfile` держит счётчики тегов за 30 дней. Режим «Охота на ошибки»: топ-3 тега → карта tag→skills (статическая таблица в коде: `article → [a1_articles_basic]`, `aux_missing → [a1_do_questions, a1_does_questions, a2_did_questions_neg]`, `verb_tense → [b1_pres_perf_vs_past, a2_past_simple_*]`, …) → сессия из этих навыков + при наличии ключа 5 персональных предложений от generator с `WEAK_TAGS`.

### 10.5 Дневной план (экран «Сегодня»)

1. Due-навыки по FSRS (максимум 3 review-сессии).
2. Пиявки (вплетаются в разминки).
3. Новый материал текущего модуля.
4. Предложение «охоты», если какой-то тег > порога.
5. Личные примеры «Про себя» (§8.4) периодически всплывают в разминках.
6. Если письмо заметно опережает речь (§11.1) — план в этот день предпочитает голосовой/Speaking-раунд новому письменному материалу.
7. Бэклог due после перерыва: разбирается от самых слабых (низкая retrievability), кап «3 review-сессии/день» сохраняется — остальные ждут (анти-«стена из 50 ревью» с порога, как в Anki).
8. С ростом уровня баланс смещается от нового переводного дрилла к спринтам/Free Talk/Speaking (на B1+ заметно чаще): доля не-переводных активностей растёт — противодействие закреплению привычки «думать через перевод» (риск №1 §17.1).

### 10.6 Прогноз памяти

FSRS даёт retrievability — вероятность вспомнить навык прямо сейчас. Показываем её на карте курса (кольцо вокруг навыка) и на «Сегодня» («Present Perfect: помнишь на ~60% — пора повторить»). Прозрачная механика мотивирует сильнее абстрактного «пора позаниматься». Честная оговорка (2026-07-08): FSRS откалиброван на атомарных карточках, наша карточка — агрегат навыка, поэтому retrievability здесь — ориентир, а не измерение: в UI показывать ступени/тильду («~60%», «пора повторить», «ещё держится»), низкие значения не красить в тревожный красный — после перерыва «всё красное» убивает мотивацию, а не подстёгивает.

## §11. Placement, скипы, экзамены (всё детерминировано, без LLM)

### 11.1 Placement (~10 мин, ≤25 предложений)

Адаптивная лестница по probe-предложениям (§3.2): старт на A2 → ступень = 5 предложений уровня (probe разных модулей) → ≥4 верно = вверх, ≤2 = вниз, 3 = зона найдена; повторное посещение ступени завершает тест. Результат: рекомендуемый стартовый модуль; всё ниже — `passed(auto)`, модули найденного уровня размечаются по 2 probe (2/2 → passed). Экран результата честный: «Похоже на B1. Начнём с М B1-1, но карта открыта — решаешь ты».

**Разрыв «читаю/пишу лучше, чем говорю».** Лестница проверяет только письменный перевод — она системно завышает уровень для тех, у кого пассивное понимание сильно опережает активную речь (частый случай, включая целевого юзера §1.1). Поэтому сразу после письменного результата — один прямой вопрос («Как оцениваешь свою РАЗГОВОРНУЮ речь: так же / заметно слабее?»). Ответ «заметно слабее» не понижает письменный уровень (грамматика реально освоена), но: (а) с первого дня врубает голосовые раунды (§3.3) и Speaking-сессии (§9.4) чаще, чем дефолт — обгонять письмо не даём; (б) дневной план (§10.5) поднимает приоритет речевых модальностей выше нового материала, пока разрыв не сократится (мерять грубо: точность письма vs точность голосовых раундов по последним 20 попыткам).

### 11.2 Скип-тест модуля (в любой момент)

8 предложений модуля (probe + difficulty 4–5), без подсказок; ≥7 → модуль `passed`. «Знал, но подзабыл» → быстро прошёл лестницу скипов и оказался на своём фронтире.

### 11.3 Экзамен модуля

16 предложений всех навыков модуля, без подсказок, одна попытка на предложение; pass ≥80%. Перед стартом — экран ожиданий: «16 предложений, без подсказок, проходной 80%; провал ничего не сжигает — покажу, что повторить». Fail → показывается разбор ошибок по тегам + рекомендация «повторить навыки X, Y», retry через 4 часа.

### 11.4 Экзамен уровня

30 предложений (все модули, взвешенно по слабым) + 6 listening-диктантов. Pass: ≥80% общий и ≥4/6 listening. **Гейт не имеет права стоять на ненадёжном TTS (2026-07-08):** перед listening-частью — калибровочная фраза («слышно чётко?»); TTS недоступен или юзер ответил «плохо» → listening-часть заменяется 6 письменными предложениями, в результате пометка «сдан без аудирования». Финальный экран-«сертификат» (canvas-картинка «A2 закрыт», сохраняется в галерею) — маленький праздник, нулевая инфраструктура.

## §12. UX и мотивация

- **Мобайл-first**, тёмная+светлая тема, минимализм, крупная типографика, всё управляется с клавиатуры на десктопе.
- **Время до первого предложения < 60 секунд** с первого захода (онбординг не мучает).
- **Streak** (дни с ≥1 сессией), **дневная цель** (default 30 предложений ≈ 15-мин сессия), **XP+combo** за серии верных ответов, **карта-«метро»** прогресса уровня, экзамен = «босс» (метафора плана; в UI — «экзамен модуля», без геймерского жаргона), бейджи (модуль/уровень/1000 предложений/7-дневный streak). Рядом со streak — невыгораемый счётчик «всего дней занятий»; потеря streak подаётся щадяще (streak — осознанный компромисс с loss aversion в мягкой конфигурации: заморозка, 5-мин спасение, ноль пуш-давления — §17.1).
- **Заморозка streak**: 1 автоматическая заморозка в неделю — пропущенный день не сжигает цепь (потеря 40-дневного streak из-за одного дня — главный убийца мотивации; 5-минутный пресет тоже «спасает день»).
- **Недельный отчёт** (локальный): генерится за прошлую неделю при первом заходе с понедельника и висит на «Сегодня» до просмотра (юзер, не зашедший в понедельник, отчёт всё равно увидит): что пройдено, топ-3 ошибки, прогноз памяти, одна рекомендация на неделю.
- **Два языка интерфейса**: русский и английский, все строки продублированы (переключатель в настройках, автоопределение при первом входе). Направление курса не меняется: стимулы всегда русские, учим английский.
- Праздновать сдержанно (конфетти только на экзаменах), никаких дарк-паттернов, пуш-спама и «заплати за жизнь». **Напоминания (исправлено 2026-07-08 — прежняя формулировка обещала нереализуемое):** настоящие push без сервера невозможны (Web Push требует push-сервиса, scheduled-уведомления в браузерах не живут, Notification Triggers API мёртв) → честный вариант: напоминание при открытии + Badging API где поддержан; настоящие пуши — только если появится Capacitor-обёртка (бэклог Ф9). В UI пуши не обещать.
- Тон UI: короткий, дружелюбный, на «ты», без канцелярита и без сюсюканья.
- Под каждым LLM-разбором — мелкий дисклеймер «ИИ может ошибаться, перепроверяй» (как у English Dog).

## §14. Хостинг, PWA, приватность

1. **Хостинг**: **GitHub Pages** (решение юзера 2026-07-07; репозиторий публичный, проект открытый и некоммерческий). Деплой из GitHub Actions по пушу в main: build app + `packs/`. Cloudflare Pages — запасной вариант, если упрёмся в лимиты. API-ключ юзера при этом хранится в localStorage его браузера (§14.3) — аккаунты и сервер не нужны.
2. **PWA**: precache — shell и `packs/index.json`; runtime-cache — паки; кнопка «Скачать весь курс» (+ модель kokoro в CacheStorage) — обе кнопки показывают размер до старта (~4 МБ курс, ~86 МБ голос) и идемпотентно докачивают недостающее при обрыве. Обновление SW: autoUpdate; консистентность паков — по checksum из `packs/index.json` (§4.8). Полный офлайн, кроме LLM/STT/YouGlish; индикатор «офлайн» в шапке (navigator.onLine), ИИ-кнопки серые с причиной. **Install-prompt**: после первой успешной сессии — мягкое «добавь на экран» (на iOS это ещё и защита данных: ITP не чистит установленные PWA, §14.6).
3. **Ключи и приватность контента**: ключи — только localStorage, никуда не отправляются кроме самого провайдера. Экран ввода ключа: «ключ живёт в этом браузере; используй ключ без привязанного биллинга — расход контролируешь ты». CSP через `<meta>` (GitHub Pages не отдаёт заголовки → без frame-ancestors/report-uri, риск кликджекинга принят; Cloudflare Pages вернул бы заголовки): `default-src 'self'` + хосты провайдеров + YouGlish iframe + **`http://localhost:* http://127.0.0.1:*` в connect-src** — иначе CSP заблокирует локальные модели §8.7 (конфликт двух секций найден 2026-07-08); LAN-IP с https не работает (mixed content) — честно в README. Никакой телеметрии/аналитики. **Страница приватности перечисляет, что уходит наружу:** контент ИИ-режимов (ответы, «Про себя», Free Talk) — твоему LLM-провайдеру; голос при облачном STT — облаку браузера (opt-in диалог, §9.3); клик по YouGlish — загрузка стороннего виджета (запрос+IP). Остальное не покидает браузер.
4. **Страница «Данные и лицензии»** (обязательная, CC-BY): Tatoeba (CC-BY 2.0 FR), поимённая атрибуция авторов предложений (агрегированный сворачиваемый список — только авторы вошедших в паки предложений, это сотни имён, не 536k), Kokoro (Apache-2.0), прочие зависимости; generated-предложения — под CC0/лицензией проекта, происхождение различает поле `source`. Лицензия output Gemini подтверждена (2026-07-08): Google не претендует на права на сгенерированное, публикация в открытом репо допустима (после ручной фильтрации/вычитки), но паки нельзя позиционировать как датасет для обучения ML-моделей (запрет Google ToS) — дисклеймер об этом в README (§18.6). Здесь же — блок «что уходит наружу» из п.3.
5. **Бэкап**: экспорт/импорт всей Dexie в JSON-файл — защита от потери IndexedDB. Файл содержит `dbVersion`; импорт старого бэкапа прогоняет штатные Dexie-миграции (тест — §15). **Ключи провайдеров в экспорт по умолчанию не попадают** (чекбокс «включить ключи», default off — файл-бэкап понесут через мессенджеры, секрету там не место). Прогресс живёт на одном устройстве; перенос = экспорт/импорт целиком (перезапись, без merge) — честно сказать в UI; «импорт с объединением» двух прогрессов — бэклог Ф9.
6. **Защита хранилища (добавлено 2026-07-08 — для SRS-продукта потеря прогресса фатальна):** (а) `navigator.storage.persist()` при первом сохранении прогресса; (б) эвристика приватного режима — аномально малая квота `storage.estimate()` → предупреждение «прогресс не сохранится после закрытия окна»; (в) Safari ITP стирает storage после ~7 дней неактивности → install-prompt на iOS обязателен (п.2) + более настойчивое напоминание о бэкапе на Safari; (г) напоминание о бэкапе: баннер на «Сегодня» раз в ~30 дней или каждые ~20 сессий; (д) мультивкладочность: Web Locks или предупреждение «приложение уже открыто в другой вкладке» (две вкладки не должны гонять запись прогресса).

## §15. Тестирование и quality gates

| Слой | Инструмент | Что покрываем |
|---|---|---|
| Checker/normalizer | Vitest, табличные тесты **≥120 кейсов** | контракции и ветвление `'s/'d` (вкл. многоапострофные), BrE/AmE (вкл. learnt/learned), цифры↔слова (вкл. «the red one»), «опечатка vs морфология» (`work/works` — НЕ опечатка), «опечатка vs другое слово» (`want/went`, `than/then`, `lose/loose` — НЕ опечатка; `teh/the` — опечатка), артикли/предлоги → error, tier 0 (кириллица/пустой ввод), кэш-хиты |
| Drill engine | Vitest | очередь, interleave, requeue, hint ladder, REWRITE-цикл (ответ скрыт → поле пусто → успех) |
| SRS | Vitest + fake timers | маппинг рейтингов, due-логика, пиявки |
| LLM-контракты | Vitest + zod + записанные фикстуры | валидные/битые JSON judge/generator, retry-ветка, «explanation_ru не на русском» → retry; фикстуры = мини-бенчмарк судьи §7.5 (60–100 размеченных кейсов) |
| E2E | Playwright | смоук: онбординг → дрилл → ошибка → разбор → REWRITE → финал; офлайн со старта И обрыв сети посреди сессии (setOffline mid-drill → каскад падает в tier 4 с баннером, обучение продолжается); замер «<60 с до первого предложения» (холодный кэш, эмуляция Fast 3G) |
| Пайплайн | pytest, golden-тесты | правила теггера (фикстуры на каждый навык), clean-фильтры, level-пороги, `etr validate` |
| CI | GitHub Actions | eslint+prettier, tsc, vitest, playwright (смоук), ruff+mypy+pytest, `etr validate`, build |

Правила: любой баг чекера → сначала кейс в таблицу тестов, потом фикс. Любое изменение схемы Dexie → новая версия + миграция + тест «апгрейд со старой БД не теряет данные» + тест импорта старого бэкапа (§14.5) — PWA обновляется тихо, юзер со старой БД получает новый код без предупреждения.

**2026-07-08 — аудит плана, v1.2 → v1.3 (реализация не начата).** Прогнан аудит на 260 вопросов (три раунда «узких мест»), внесено ~45 правок по всему плану.
